完整配置指南 - 从入门到精通
📚 基于官方文档 v2026.3

🎯什么是多 Agent 架构

目标：在单个运行的 Gateway 中实现多个隔离的 Agent（独立的工作空间 + agentDir + 会话），以及多个渠道账号（例如两个 WhatsApp），通过绑定（bindings）将入站消息路由到不同的 Agent。

什么是"一个 Agent"？

一个 Agent 是一个完全独立的"大脑"，拥有自己的：

  • 工作空间 文件、AGENTS.md/SOUL.md/USER.md、本地笔记、角色规则
  • 状态目录 (agentDir) 用于认证配置文件、模型注册表、每个 Agent 的配置
  • 会话存储 聊天记录 + 路由状态，位于 ~/.openclaw/agents/<agentId>/sessions

⚠️ 重要：认证配置文件是每个 Agent 独立的。每个 Agent 从自己的 ~/.openclaw/agents/<agentId>/agent/auth-profiles.json 读取。主 Agent 的凭证不会自动共享。切勿在多个 Agent 之间重用 agentDir（会导致认证/会话冲突）。

🔑核心概念术语

🧠 agentId

一个"大脑"（工作空间、每个 Agent 的认证、每个 Agent 的会话存储）

📱 accountId

一个渠道账号实例（例如 WhatsApp 账号 "personal" vs "biz"）

🎛️ binding

通过 (channel, accountId, peer) 将入站消息路由到 agentId，可选包含 guild/team id

直接聊天会折叠到 agent:<agentId>:<mainKey>（每个 Agent 的"主会话"，session.mainKey）

👥多 Agent = 多角色、多性格

使用多 Agent时，每个 agentId 成为一个完全隔离的角色：

  • 不同的电话号码/账号 每个渠道 accountId
  • 不同的性格 每个 Agent 的工作空间文件（如 AGENTS.md 和 SOUL.md）
  • 独立的认证 + 会话 除非明确启用，否则不会跨 Agent 交流

这让多个人可以共享一个 Gateway 服务器，同时保持他们的 AI"大脑"和数据隔离。

🚀快速开始指南

创建每个 Agent 的工作空间

使用向导或手动创建工作空间：

创建 Agent 命令
openclaw agents add coding openclaw agents add social

每个 Agent 获得自己的工作空间，包含 SOUL.md、AGENTS.md 和可选的 USER.md，以及专用的 agentDir 和会话存储（位于 ~/.openclaw/agents/<agentId>）。

为每个 Agent 创建渠道账号

为你选择的渠道为每个 Agent 创建一个账号：

  • Discord: 每个 Agent 一个 bot，启用 Message Content Intent，复制每个 token
  • Telegram: 通过 BotFather 为每个 Agent 创建一个 bot，复制每个 token
  • WhatsApp: 为每个账号链接每个电话号码
登录渠道账号
openclaw channels login --channel whatsapp --account work
添加 Agent、账号和绑定

在 agents.list 下添加 Agent，在 channels.<channel>.accounts 下添加渠道账号，并使用 bindings 连接它们（详见配置示例页面）。

重启并验证
验证命令
openclaw gateway restart openclaw agents list --bindings openclaw channels status --probe

🛠️Agent 助手命令

添加新 Agent

openclaw agents add work

使用 Agent 向导添加新的隔离 Agent

列出所有 Agent

openclaw agents list --bindings

验证 Agent 配置和绑定

渠道状态

openclaw channels status --probe

检查渠道连接状态

📱单 WhatsApp 号码，多人使用

你可以在一个 WhatsApp 账号上路由不同的 WhatsApp 私信到不同的 Agent。使用 peer.kind: "direct" 匹配发送者 E.164（如 +15551234567）。回复仍然来自同一个 WhatsApp 号码（没有每个 Agent 的发送者身份）。

重要细节：直接聊天会折叠到 Agent 的主会话键，所以真正的隔离需要每人一个 Agent。

🎯路由规则详解

绑定是确定性的，采用最具体优先原则。

路由优先级（从高到低）

优先级 匹配类型 说明
1 peer 匹配 精确的 DM/群组/渠道 ID
2 parentPeer 匹配 线程继承
3 guildId + roles Discord 角色路由
4 guildId Discord 服务器
5 teamId Slack 团队
6 accountId 匹配 渠道账号匹配
7 渠道级匹配 accountId: "*"
8 回退到默认 Agent agents.list[].default 或第一个条目（默认：main）

⚠️ 如果同一层级有多个绑定匹配，配置顺序中的第一个获胜。

⚠️ 如果绑定设置多个匹配字段（例如 peer + guildId），所有指定字段都是必需的（AND 语义）。

📊账号作用域详情

  • 省略 accountId 的绑定仅匹配默认账号
  • 使用 accountId: "*" 实现跨所有账号的渠道级回退
  • 如果之后为同一个 Agent 添加相同绑定但带有明确的账号 ID，OpenClaw 会将现有的渠道级绑定升级为账号作用域，而不是复制

🌐支持多账号的渠道

支持多账号的渠道使用 accountId 标识每个登录。每个 accountId 可以路由到不同的 Agent，因此一个服务器可以托管多个电话号码而不会混合会话。

如果你想为省略 accountId 时设置渠道级默认账号，设置 channels.<channel>.defaultAccount（可选）。未设置时，OpenClaw 回退到 default（如果存在），否则使用第一个配置的账号 ID（排序后）。

支持的渠道列表

主流渠道

whatsapp, telegram, discord, slack, signal, imessage

IRC/聊天

irc, line, googlechat, mattermost, matrix, nextcloud-talk

其他

bluebubbles, zalo, zalouser, nostr, feishu

📝Discord 每个 Agent 一个 Bot

每个 Discord Bot 账号映射到唯一的 accountId。将每个账号绑定到一个 Agent，并为每个 Bot 保留白名单。

openclaw.json 配置示例
{ "agents": { "list": [ { "id": "main", "workspace": "~/.openclaw/workspace-main" }, { "id": "coding", "workspace": "~/.openclaw/workspace-coding" } ] }, "bindings": [ { "agentId": "main", "match": { "channel": "discord", "accountId": "default" } }, { "agentId": "coding", "match": { "channel": "discord", "accountId": "coding" } } ], "channels": { "discord": { "groupPolicy": "allowlist", "accounts": { "default": { "token": "DISCORD_BOT_TOKEN_MAIN", "guilds": { "123456789012345678": { "channels": { "222222222222222222": { "allow": true, "requireMention": false } } } } }, "coding": { "token": "DISCORD_BOT_TOKEN_CODING", "guilds": { "123456789012345678": { "channels": { "333333333333333333": { "allow": true, "requireMention": false } } } } } } } } }

注意：

  • 将每个 Bot 邀请到服务器并启用 Message Content Intent
  • Token 存放在 channels.discord.accounts.<id>.token（默认账号可使用 DISCORD_BOT_TOKEN）

📱WhatsApp 每个 Agent 一个号码

在启动 Gateway 之前链接每个账号：

登录命令
openclaw channels login --channel whatsapp --account personal openclaw channels login --channel whatsapp --account biz
openclaw.json 配置
{ "agents": { "list": [ { "id": "home", "default": true, "name": "Home", "workspace": "~/.openclaw/workspace-home", "agentDir": "~/.openclaw/agents/home/agent" }, { "id": "work", "name": "Work", "workspace": "~/.openclaw/workspace-work", "agentDir": "~/.openclaw/agents/work/agent" } ] }, "bindings": [ { "agentId": "home", "match": { "channel": "whatsapp", "accountId": "personal" } }, { "agentId": "work", "match": { "channel": "whatsapp", "accountId": "biz" } }, // 可选的每个 peer 覆盖（示例：将特定群组发送到 work Agent） { "agentId": "work", "match": { "channel": "whatsapp", "accountId": "personal", "peer": { "kind": "group", "id": "1203630...@g.us" } } } ], "tools": { "agentToAgent": { "enabled": false, "allow": ["home", "work"] } }, "channels": { "whatsapp": { "accounts": { "personal": {}, "biz": {} } } } }

💬示例：WhatsApp 日常聊天 + Telegram 深度工作

按渠道拆分：将 WhatsApp 路由到快速的日常 Agent，将 Telegram 路由到 Opus Agent。

按渠道路由
{ "agents": { "list": [ { "id": "chat", "name": "Everyday", "workspace": "~/.openclaw/workspace-chat", "model": "anthropic/claude-sonnet-4-6" }, { "id": "opus", "name": "Deep Work", "workspace": "~/.openclaw/workspace-opus", "model": "anthropic/claude-opus-4-6" } ] }, "bindings": [ { "agentId": "chat", "match": { "channel": "whatsapp" } }, { "agentId": "opus", "match": { "channel": "telegram" } } ] }

🎯示例：同一渠道，单个 Peer 路由到 Opus

保持 WhatsApp 在快速 Agent 上，但将一个私信路由到 Opus：

Peer 绑定优先
{ "agents": { "list": [ { "id": "chat", "name": "Everyday", "workspace": "~/.openclaw/workspace-chat", "model": "anthropic/claude-sonnet-4-6" }, { "id": "opus", "name": "Deep Work", "workspace": "~/.openclaw/workspace-opus", "model": "anthropic/claude-opus-4-6" } ] }, "bindings": [ { "agentId": "opus", "match": { "channel": "whatsapp", "peer": { "kind": "direct", "id": "+15551234567" } } }, { "agentId": "chat", "match": { "channel": "whatsapp" } } ] }

Peer 绑定始终获胜，所以将它们保持在渠道级规则之上。

👨‍👩‍👧‍👦示例：家庭 Agent 绑定到 WhatsApp 群组

将专用的家庭 Agent 绑定到单个 WhatsApp 群组，具有提及门控和更严格的工具策略：

家庭 Agent 配置
{ "agents": { "list": [ { "id": "family", "name": "Family", "workspace": "~/.openclaw/workspace-family", "identity": { "name": "Family Bot" }, "groupChat": { "mentionPatterns": ["@family", "@familybot", "@Family Bot"] }, "sandbox": { "mode": "all", "scope": "agent" }, "tools": { "allow": [ "exec", "read", "sessions_list", "sessions_history", "sessions_send", "sessions_spawn", "session_status" ], "deny": ["write", "edit", "apply_patch", "browser", "canvas", "nodes", "cron"] } } ] }, "bindings": [ { "agentId": "family", "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "120363999999999999@g.us" } } } ] }

注意：工具允许/拒绝列表针对的是tools，而不是 skills。如果 skill 需要运行二进制文件，确保 exec 被允许并且二进制文件存在于沙盒中。

🔒Sandbox 沙盒配置

OpenClaw 可以在沙盒后端内运行工具以减少爆炸半径。这是可选的，通过配置控制（agents.defaults.sandbox 或 agents.list[].sandbox）。如果沙盒关闭，工具在主机上运行。

什么会被沙盒化

  • 工具执行（exec, read, write, edit, apply_patch, process 等）
  • 可选的沙盒化浏览器（agents.defaults.sandbox.browser）

不被沙盒化：

  • Gateway 进程本身
  • 任何明确允许在主机上运行的工具（如 tools.elevated）
  • ⚠️ Elevated exec 在主机上运行并绕过沙盒！

⚙️沙盒模式 (Modes)

agents.defaults.sandbox.mode 控制何时使用沙盒：

"off"

不沙盒化

"non-main"

仅沙盒化非主会话（默认）

"all"

每个会话都在沙盒中运行

⚠️ 注意："non-main" 基于 session.mainKey（默认 "main"），而不是 Agent ID。群组/渠道会话使用自己的键，因此被视为 non-main 并会被沙盒化。

📦沙盒作用域 (Scope)

agents.defaults.sandbox.scope 控制创建多少个容器：

"session"

（默认）每个会话一个容器

"agent"

每个 Agent 一个容器

"shared"

一个容器由所有沙盒化会话共享

🖥️沙盒后端 (Backend)

agents.defaults.sandbox.backend 控制哪个运行时提供沙盒：

后端 运行位置 设置 最佳用途
Docker
默认		 本地容器 scripts/sandbox-setup.sh 本地开发，完全隔离
SSH 任何 SSH 可访问的主机 SSH 密钥 + 目标主机 卸载到远程机器
OpenShell OpenShell 管理的沙盒 启用 OpenShell 插件 管理的远程沙盒，可选双向同步

📁工作空间访问 (Workspace Access)

agents.defaults.sandbox.workspaceAccess 控制沙盒可以看到什么：

"none"

（默认）工具看到沙盒工作空间下的 ~/.openclaw/sandboxes

"ro"

只读挂载 Agent 工作空间到 /agent（禁用 write/edit/apply_patch）

"rw"

读写挂载 Agent 工作空间到 /workspace

🔧自定义绑定挂载

agents.defaults.sandbox.docker.binds 将额外的主机目录挂载到容器中。格式：host:container:mode（如 "/home/user/source:/source:ro"）。

绑定挂载示例
{ "agents": { "defaults": { "sandbox": { "docker": { "binds": [ "/home/user/source:/source:ro", "/var/data/myapp:/data:ro" ] } } }, "list": [ { "id": "build", "sandbox": { "docker": { "binds": ["/mnt/cache:/cache:rw"] } } } ] } }

安全注意：

  • 绑定绕过沙盒文件系统：它们以你设置的任何模式（:ro 或 :rw）暴露主机路径
  • OpenClaw 阻止危险的绑定源（如 docker.sock, /etc, /proc, /sys, /dev）
  • 敏感挂载（密钥、SSH 密钥、服务凭证）应该是 :ro，除非绝对必要

📦镜像 + 设置

默认 Docker 镜像：openclaw-sandbox:bookworm-slim

构建沙盒镜像
# 基础沙盒镜像 scripts/sandbox-setup.sh # 功能更完整的沙盒镜像（包含 curl, jq, nodejs, python3, git） scripts/sandbox-common-setup.sh # 沙盒浏览器镜像 scripts/sandbox-browser-setup.sh

注意：默认镜像不包含 Node。如果 skill 需要 Node（或其他运行时），要么构建自定义镜像，要么通过 sandbox.docker.setupCommand 安装（需要网络出口 + 可写根目录 + root 用户）。

🛠️每 Agent 沙盒和工具配置

多 Agent 设置中的每个 Agent 都可以覆盖全局沙盒和工具策略。本页涵盖每 Agent 配置、优先级规则和示例。

📊配置优先级

当全局（agents.defaults.*）和 Agent 特定（agents.list[].*）配置都存在时：

沙盒配置

Agent 特定设置覆盖全局：

agents.list[].sandbox.mode > agents.defaults.sandbox.mode agents.list[].sandbox.scope > agents.defaults.sandbox.scope agents.list[].sandbox.workspaceRoot > agents.defaults.sandbox.workspaceRoot agents.list[].sandbox.workspaceAccess > agents.defaults.sandbox.workspaceAccess agents.list[].sandbox.docker.* > agents.defaults.sandbox.docker.* agents.list[].sandbox.browser.* > agents.defaults.sandbox.browser.* agents.list[].sandbox.prune.* > agents.defaults.sandbox.prune.*

工具限制

过滤顺序：

  1. 工具配置文件 (tools.profile 或 agents.list[].tools.profile)
  2. 提供者工具配置文件 (tools.byProvider[provider].profile)
  3. 全局工具策略 (tools.allow / tools.deny)
  4. 提供者工具策略 (tools.byProvider[provider].allow/deny)
  5. Agent 特定工具策略 (agents.list[].tools.allow/deny)
  6. Agent 提供者策略 (agents.list[].tools.byProvider[provider].allow/deny)
  7. 沙盒工具策略 (tools.sandbox.tools 或 agents.list[].tools.sandbox.tools)
  8. 子 Agent 工具策略 (tools.subagents.tools，如果适用)

每个级别可以进一步限制工具，但不能恢复之前级别拒绝的工具。

📝工具限制示例

📖 只读 Agent

{ "tools": { "allow": ["read"], "deny": ["exec", "write", "edit", "apply_patch", "process"] } }

✅ 安全执行 Agent

（无文件修改）

{ "tools": { "allow": ["read", "exec", "process"], "deny": ["write", "edit", "apply_patch", "browser", "gateway"] } }

💬 仅通信 Agent

{ "tools": { "sessions": { "visibility": "tree" }, "allow": [ "sessions_list", "sessions_send", "sessions_history", "session_status" ], "deny": ["exec", "write", "edit", "apply_patch", "read", "browser"] } }

👨‍👩‍👧‍👦 家庭 Agent

{ "sandbox": { "mode": "all", "scope": "agent" }, "tools": { "allow": ["exec", "read", "sessions_*", "session_status"], "deny": ["write", "edit", "apply_patch", "browser", "canvas", "nodes", "cron"] } }

📋示例：个人 + 受限家庭 Agent

完整配置示例
{ "agents": { "list": [ { "id": "main", "default": true, "name": "Personal Assistant", "workspace": "~/.openclaw/workspace", "sandbox": { "mode": "off" } }, { "id": "family", "name": "Family Bot", "workspace": "~/.openclaw/workspace-family", "sandbox": { "mode": "all", "scope": "agent" }, "tools": { "allow": ["read"], "deny": ["exec", "write", "edit", "apply_patch", "process", "browser"] } } ] }, "bindings": [ { "agentId": "family", "match": { "provider": "whatsapp", "accountId": "*", "peer": { "kind": "group", "id": "120363424282127706@g.us" } } } ] }

结果：

  • main Agent: 在主机上运行，完全工具访问
  • family Agent: 在 Docker 中运行（每个 Agent 一个容器），只有 read 工具

⚠️常见陷阱："non-main"

agents.defaults.sandbox.mode: "non-main" 基于 session.mainKey（默认 "main"），而不是 Agent ID。

群组/渠道会话总是获得自己的键，因此它们被视为 non-main 并会被沙盒化。

如果你想让一个 Agent 从不沙盒化，设置 agents.list[].sandbox.mode: "off"。

🐛常见问题

Agent 没有沙盒化，尽管设置了 mode: "all"

  • 检查是否有全局 agents.defaults.sandbox.mode 覆盖它
  • Agent 特定配置优先，所以设置 agents.list[].sandbox.mode: "all"

尽管有拒绝列表，工具仍然可用

  • 检查工具过滤顺序：全局 → Agent → 沙盒 → 子 Agent
  • 每个级别只能进一步限制，不能恢复
  • 使用日志验证：[tools] filtering tools for agent:${agentId}

容器没有每个 Agent 隔离

  • 在 Agent 特定沙盒配置中设置 scope: "agent"
  • 默认是 "session"，会为每个会话创建一个容器

🔄从单 Agent 迁移

之前（单 Agent）

{ "agents": { "defaults": { "workspace": "~/.openclaw/workspace", "sandbox": { "mode": "non-main" } } }, "tools": { "sandbox": { "tools": { "allow": ["read", "write", "apply_patch", "exec"], "deny": [] } } } }

之后（多 Agent）

{ "agents": { "list": [ { "id": "main", "default": true, "workspace": "~/.openclaw/workspace", "sandbox": { "mode": "off" } } ] } }

遗留的 agent.* 配置会被 openclaw doctor 迁移；以后优先使用 agents.defaults + agents.list。
