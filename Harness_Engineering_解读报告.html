📋 一、核心概念

Harness Engineering（套具工程） 是 2025 年底到 2026 年初，AI 工程领域崛起速度最快的概念。它解决的核心问题是：为什么模型能力越来越强，但实际应用的 Agent 还是经常"翻车"？

💡 核心定义：Harness Engineering 不是在教模型怎么回答，而是在设计模型怎么工作。它处理的是模型外部那一层东西——任务怎么拆解、上下文怎么管理、工具怎么编排、权限怎么设定、状态怎么交接、做完了怎么验证、失败了怎么恢复、什么时候该把控制权交回给人类。

🐴 Harness 的词源："Harness"原意是马具（缰绳、马鞍、嚼子等一整套控制马匹的装备）。这个隐喻是刻意的——马是模型（强大、快速，但自己不知道该往哪跑），骑手是人类工程师（提供方向），而马具就是 Harness（把这批马的原始能力导向有用的工作）。

📈 二、技术演进路线

三个概念形成了一条非常清晰的演进路线：

层级 概念 主导时期 核心问题 关注点
第一层 Prompt Engineering 2020-2024 怎么问 怎么措辞、怎么给示例、要不要用思维链
第二层 Context Engineering 2025 开始 让模型看到什么 RAG、技术栈、工具定义、对话历史管理
第三层 Harness Engineering 2026 开始 让模型在什么机制里干活 工具编排、权限设计、状态保持、验证机制、日志、失败恢复、人工介入
🚗 精彩比喻：Prompt 是"右转"这个命令；Context 是给模型一张地图让它理解右转是什么意思；而Harness 是整辆车——方向盘、刹车、车道边界、维护计划、警示灯，以及确保车门不会在高速公路上脱落的所有工程设计。

包含关系：Harness ⊃ Context ⊃ Prompt

🔥 三、为什么是 2025 年底到 2026 年初爆发

原因 1：模型能力抬高后，系统设计变成主要差异来源
说白了就是模型够强了，但光强没用，你得给它一个好的工作环境，它才能把能力发挥出来。
原因 2：长时任务暴露了纯模型的系统性缺陷
即使是 Opus 4.5，在跨多个上下文窗口运行时，如果只给一个高层级指令（比如"构建一个可靠的 AI 应用"），它依然造不出生产级的 Web 应用。失败模式非常典型：要么试图一口气完成所有事情导致上下文耗尽，要么下一个三反看到一部分进展就提前宣布完成而不验证。
原因 3：数学直觉说明问题的严重性
假设一个多步 Agent 流水线中，每一步的成功率是 95%（听起来很高了），但串联 20 步之后，整体的任务完成率只剩 36%。这就是为什么团队报告说 Agent 95% 的时间都在正常工作，但真实任务上仍然有 1/3 的失败率。
原因 4：模型正在趋向商品化
GPT 系列、Claude 系列、Gemini 系列在核心能力上的差距在缩小。当模型本身不再是差异化因素的时候，"让模型工作的系统设计"也就是 Harness 就成了新的竞争壁垒。

⚠️ 关键判断：旧护城河是模型质量，新护城河是 Harness 质量。

🏢 四、头部公司实践案例

1. Anthropic（Claude）

📅 2025 年 11 月 - 双 Agent 架构

初始化 Agent：只在第一个 Session 运行，负责建立环境、创建初始化脚本、写入进度日志文件、建立 Git 仓库，然后把用户的高层级指令扩展成数百条具体的、可测试的功能需求清单（JSON 格式）。

Bema Agent：在后续 Session 中工作，每次启动先确认当前位置、读进度文件、审查功能清单、跑已有测试。

核心思想：外部制品成为 Agent 记忆。

📅 2026 年 3 月 - 三 Agent 架构

Planner：负责扩展需求

Generator：负责实现

Evaluator：负责用 Playwright 这类工具做交互式验证和打分

关键发现：工程化一个独立的、严格的评估器 Agent，比让生成器 Agent 自我批评要容易得多。

2. OpenAI（Codex）

📊 具体数字：2025 年 8 月开始，一个最初只有 3 个人（后来扩展到 7 人）的工程团队，用 GPT-5 驱动的 Codex Agent，在大约 5 个月里生成了约100 万行代码，合并了约1500 个 PR，构建了一个有内部日活用户和外部 Alpha 测试者的生产级产品。团队人均日吞吐量大约 3.5 个 PR，而且随着团队扩大吞吐量反而上升了。

约束：零行手写代码——所有应用逻辑、测试、CI/CD、文档、可观测性和内部工具全部由 Codex 生成。

三大支柱（Morgan Flodler 概括）：

  • Context Engineering：持续增强代码库中的知识库，加上 Agent 对可观测性数据和流量器的动态访问
  • 架构约束：不只是靠 Agent 自觉遵守，而是用确定性的自定义 Linter 和结构测试来执行
  • 熵回收：定期运行后台 Agent 扫描文档不一致和架构违规，对抗系统的熵增

3. Google DeepMind（AlphaProof）

三组件 Agentic Harness：

  • Generator：负责提出候选解法和证明策略
  • Verifier：用自然语言检查逻辑缺陷和幻觉
  • Reviser：负责修正验证器发现的错误

3 个组件循环迭代，直到输出通过验证。

🎯 注意到：Generator → Verifier → Reviser，这跟 Anthropic 的 Planner → Generator → Evaluator 三 Agent 架构高度对应。两家公司独立走到了同一个设计模式上，这说明生成 - 评估分离正在成为 Agent Harness 设计的行业共识。

🧩 五、成熟 Harness 的 6 大核心模块

1️⃣ 上下文工程与制品管理

  • 项目指令文件（Agent.md、Code.md）
  • 动态上下文注入（从日志、指标、追踪信息中获取实时信息）
  • 上下文隔离（用子 Agent 作为上下文防火墙）
  • 上下文压缩（随着窗口被填满自动丢弃或摘要无关信息）

2️⃣ 工具编排与权限设计

  • 精心裁剪工具集（不是越多越好）
  • 移除冗余选项
  • MCP 协议集成
  • 文件系统访问管理、沙箱隔离

3️⃣ 验证机制与硬约束

  • 确定性约束（自定义 Linter、结果测试、Pre-commit Hooks）
  • 生成 - 评估分离
  • 自动审查循环

4️⃣ 状态管理与记忆持续性

  • 进度追踪文件
  • 结构化功能清单
  • 增量 Git 提交
  • 检查点与恢复机制

5️⃣ 可观测性与反馈闭环

  • 执行追踪
  • 质量分级
  • 异常检测
  • 反推归因（把 Agent 在生产中的失败模式追溯到 Harness 的具体缺陷）

6️⃣ 人类介入与生命周期管理

  • 关键决策点暂停（要删数据库、要扣费、要发客户邮件必须让人类确认）
  • 升级路径
  • 失败重试
  • 完整的生命周期公示

💭 六、是新瓶装旧酒吗？

坦白说，Harness Engineering 理论里确实有很大一部分不是新发明：

  • Test Harness 在软件工程里已经有几十年历史
  • CI/CD 管道、Test-Driven Development 是成熟的 DevOps 实践
  • 任务分解与编排在分布式系统里早就被充分研究
  • 沙箱隔离是安全工程的基础概念
  • 可观测性在 SRE 领域已经高度成熟
✨ 但 Harness Engineering 确实在几个维度上产生了新的方法论贡献：
  1. 约束对象发生了根本变化：传统软件工程约束的是确定性代码执行，而 Harness Engineering 约束的是概率性推理系统
  2. "约束以提升"的反直觉原则：约束 Agent 的解决空间（减少工具、限定模式、强制架构编辑）反而提升了它的生产力和可靠性
  3. 生成 - 评估分离模式：虽然在 AI 架构中灵感可以追溯到 Gan 架构，但在 Agent 系统中被重新发现
  4. 代码库本身成为 Harness 的一部分：代码结构、命名约定、模块边界不仅服务于人类可读性，更服务于 Agent 的可推理性

📌 定位：Harness Engineering 不是纯粹的新瓶装旧酒，但也不是从零开始的全新学科。它更准确的定位是——在 Agent 生产化的压力下，将多个成熟工程领域的实践重新组合、调试，并补充针对概率性推理系统的新模式，形成了一个统一的方法论。

⚠️ 七、风险与局限

风险 1：概念膨胀
Harness Engineering 正在变成一个什么都能往里装的筐。当一个术语从 Agent CMD 文件到完整的生产运营系统都能涵盖时，它的精确性和可用性就会被稀释。
风险 2：过度工程化
OpenAI 自己都强调 Harness 必须是可思列的。随着模型变强，之前需要复杂 Harness 来补偿的问题可能被模型直接解决。Anthropic 已经给出了实证——Opus 4.5 自己消除了 Sonnet 4.5 的上下文交替行为，早期 Harness 里的上下文重置机制就成了多余的。
风险 3：证据基础薄弱
支持 Harness Engineering 价值的大多数证据来自 AI 工具厂商自身（OpenAI 报告 Codex 有多厉害、Anthropic 报告 Claude Agent SDK 改进了多少、Google 报告自己的框架有多好）。这些来源都存在利益冲突。独立的、定量的、可复现的 Benchmark 验证目前仍然缺乏。
风险 4：可复现性缺口
OpenAI 的百万行代码案例是在极其特定的条件下完成的（从零开始、用自家的 Codex 工具、团队本身就是 AI 系统专家）。这个经验对普通工程团队的可复现性完全没被验证过。
风险 5：风险放大
Anthropic 在 Benchmark 的复盘中发现，多 Agent 的配置并不一定改变模型想走捷径的倾向，但因为更高的 Token 使用量和更多并行搜索者，反而提高了意外污染的概率。单 Agent 配置下非预期解法的发生率是 10.4%，多 Agent 配置下上升到 10.8%。Harness 不只是性能放大器，也可能是风险放大器。

🎯 八、务实行动路线

核心结论三句话：

  • Prompt Engineering 解决怎么说
  • Context Engineering 解决给模型看什么
  • Harness Engineering 解决让模型在什么运行机制里干活，并且如何确保它正地把活干成

🚀 立即能做

📈 中期投入

🏗️ 长期要做

设计模块化的、可替换的 Harness 架构，支持模型升级时平滑迁移。
